Controlled-source research agent

Research you can replay.

Argus produces scheduled monitoring briefs and on-demand research reports from a human-approved source registry and your own local corpus — never from a search engine's ranking. Every sentence it ships cites an immutable, content-hashed snapshot; every claim carries an explicit corroboration verdict; every run emits a manifest an auditor can re-execute.

01Purpose

Why this system exists

Off-the-shelf web research agents delegate source selection to a search engine. Ranking optimizes relevance and freshness — not authority — and it re-ranks nondeterministically between runs: a daily query cites a wire service one morning and an anonymous blog the next, with no way to constrain, audit, or reproduce the choice. Argus inverts this: source selection is a governance decision made by humans in a git-reviewed registry, and the runtime can only ever work inside it.

G1 · Source control

Only sources on the tiered allowlist can be fetched or cited. There is structurally no other fetch path.

G2 · Determinism

Same registry commit + snapshot set + config ⇒ a replayable run with stable, auditable citations.

G3 · Verifiability

Every claim traces to content-hashed snapshots; corroboration status is explicit, never implied.

G4 · Hybrid corpora

Web feeds and your local folder flow through one ingestion path, one index, one retrieval API.

G5 · Domain genericity

The engine contains zero domain code. A “domain” is a profile: tags, dictionaries, policy, templates.

G6 · Auditability

Each run emits a manifest answering: which sources, what they said, why each claim got its verdict.

G7 · Human governance

New sources queue for human review; the registry changes only via reviewed git commits.

Non-goals

No open-web crawling, no “objective truth” arbitration, no trading signals, no paywall workarounds.

Two workloads

Daily brief — fixed pipeline over a time window. Deep research — planned, bounded retrieval over the full index.

The two invariants — hold everywhere

1 · Nothing off-registry is ever fetched for evidence. Gated discovery may observe the open web, but its findings terminate at the candidate queue as proposals — never snapshots, never citations.

2 · Trust policy executes in exactly one place. Both workloads attach the same corroborate → synthesize → verify tail (graphs/shared.py), so a brief can never bypass a rule the research graph respects.

02Architecture

Three planes, one direction of trust

The system separates who decides what to trust (governance, human-paced), how material is acquired and indexed (data, scheduled), and how a run turns evidence into cited output (runtime, per invocation). Trust flows downward only: the registry configures the fetchers; the index feeds the graphs; the graphs can send proposals back up — never evidence.

Governance plane

registry/sources.yaml is the only place trust is assigned: per-tag tiers (a wire can be tier 1 for news, tier 3 for commodities), a mandatory human-readable license, an independence_group for syndication families, and an active / paused / retired switch. Changes are reviewed git commits; the commit hash is stamped on every fetch event and run.

Data plane

Adapters are deliberately dumb — fetch, don't judge. Everything fetched is frozen as a content-addressed snapshot before any processing, then an idempotent LangGraph ingestion pass extracts text, clusters syndicated copies (MinHash), tags entities from domain dictionaries, chunks, embeds, and upserts to Qdrant.

Runtime plane

Two graphs, one tail. The daily graph is a deterministic pipeline over a time window; the research graph adds a planner and a bounded retrieve⇄reflect loop. Both hand their evidence to the same corroborate → synthesize → verify nodes, checkpointed per superstep under thread_id == run_id.

src/argus/claims/
The trust core

Extractors (stub regex / Azure structured-output) emit event and quantitative claims born citing their snapshot. The engine is deterministic policy code: union-find voice counting over cluster_id OR independence_group, tier thresholds, tolerance matching, and four verdicts. src/argus/graphs/
Workload orchestration

src/argus/synthesis/
Grounded writing, enforced citations

Synthesizers see only verdict-annotated claims and their snippets — never the open index. Fabricated [S#] markers are stripped; every sentence must then carry a real citation or it fails verification and rides the retry-then-drop loop. Jinja templates add the sources table and provenance footer.

src/argus/governance/
Proposals, not shortcuts

Discovery hits become candidates rows deduped by domain name; rejected domains are never re-proposed. Approval does not edit the registry — it prints a paste-ready YAML stanza born paused with the license unfilled, because the registry changes only via reviewed commits.

src/argus/observe/
Quality management

health_report rolls fetch events into per-source status (ok / degraded / failing / unknown), consecutive failures, and new-snapshot yield — the early warning for feed changes. run_golden executes canned cases through the offline claims pipeline and scores verdict expectations plus citation precision against per-domain targets, in CI.

05Workflows

How a run actually moves

05·A — The daily brief

A deterministic pipeline: no planner, no agentic search — just “what entered the window, corroborated and cited.” Triggered by the console’s scheduler or Run-now button (§09), by cron, or by hand as argus brief --domain <name>; the profile's brief_window_hours (default 24) defines “new.”

05·B — Deep research

On demand — argus research "question" --domain <name>, or the console’s Ask-the-archive box (§09). A planner decomposes the question (≤ 4 sub-questions), gated discovery may propose new sources, then a bounded retrieve⇄reflect loop gathers evidence across the full index — web snapshots and local corpus alike — before the same shared tail runs.

05·C — Corroboration: how a verdict is earned

The engine is deterministic policy code, not an LLM. It groups matching claims (events by pairing; quantities by entity · metric · as-of date with values compared under the profile's tolerance), collapses dependent evidence into voices, then applies the domain's thresholds.

VerdictAwarded whenRendered in the brief as
confirmedvoices ≥ min_sources and tier-1 voices ≥ min_tier1…(confirmed by N independent sources). [S1][S4]
attributedbelow threshold and fallback: attribute — the default posture for thin domains (R4)…(reported by gulf_energy, unconfirmed). [S3]
contradictedevents: assertion + denial paired into one group · quantities: value groups outside tolerance…(CONFLICTING REPORTS — asserted by X; denied by Y). [S1][S2]
insufficientbelow threshold with fallback: flag; with fallback: drop the claim is excluded from synthesis but preserved in the manifest for audit…(unverified). [S5]

05·D — Replay & audit

argus replay <run_id> loads the run's manifest and re-executes the full claims tail against the frozen evidence set — no polling, no re-retrieval. Byte-identical output is not promised (LLM nondeterminism); evidence-identical output is: same snapshots, same citable universe. The command prints the original vs. replay citation sets side by side, and the manifest holds everything an auditor needs — snapshot hashes with full chunk text, every claim with its verdict and voice counts, verification stats, model fingerprints, the registry commit, and the profile hash.

Tiers are per-tag, and they never touch ranking

A tier is authority within a topic: TIER 1 primary / wire / official (can confirm), TIER 2 reputable editorial, TIER 3 supplementary (may inform, may be quoted as attributed, never sufficient alone). The same outlet carries different tiers per tag — in the shipped registry, Al Jazeera is tier 2 for general_news but tier 3 for commodities. Retrieval attaches the tier to every hit and ranks by relevance only; keeping ranking and trust orthogonal means a tier-3 source can still surface a story the corroborator then refuses to confirm — including the disagreeing evidence contradictions are made of.

{

  "claims": [ {text, type, verdict, independent_voices, tier1_voices, sids, conflict?, dropped?}, … ],
  "verdict_counts": {"confirmed": 4, "attributed": 17, …},

Because evidence carries the full chunk text, the manifest alone is sufficient to replay the tail and to answer the §12.2 audit question — claim → verdict → snapshots → sources → tiers — without touching the network.

07Reference

Operating Argus

Quickstart — offline in five minutes

The stub stack needs no Azure account and no network: deterministic extraction, heuristic pairing and entailment, hashing embeddings. The trust machinery — registry scoping, snapshots, clustering, verdicts, citation enforcement, manifests, replay — is identical to production.

$ pip install -e ".[dev]"
$ export ARGUS_LLM=stub ARGUS_EMBEDDER=hashing

$ argus validate                          # registry + adapters + every profile; exit 1 on problems
$ argus poll --domain general_news        # fetch in-scope sources → content-addressed snapshots
$ argus ingest                            # pending snapshots → extract, cluster, tag, chunk, index
$ argus search "hormuz war-risk premium" --domain general_news
$ argus brief --domain general_news       # the whole daily graph → brief.md + manifest.json
$ argus replay 20260705-120643-e054b7     # evidence-identical re-run from the manifest
$ argus research "How are Hormuz transits evolving?" --domain commodities
$ argus health                            # per-source liveness + yield over the last 24 h
$ argus golden --domain general_news      # verdict expectations + citation precision vs targets

…or the same, from a browser

$ pip install -e ".[webapp]" && argus-web     # http://127.0.0.1:8765 — every command below has a console home (§09·G)

Environment variables (prefix ARGUS_)

Runtime settings are environment-driven, with one loading door: get_settings() first folds two optional files into the process environment (argus/envfile.py), then pydantic-settings parses the ARGUS_* keys and the model factory reads AZURE_OPENAI_* from os.environ. Declare values in your shell, in a project-root .env (copy the annotated .env.example; gitignored; fills gaps, never overrides an export), or in the console's Settings page (§09·D) — precedence, lowest to highest: defaults < .env < shell < console overrides. Secrets never live in versioned config files. Empty values derive sensible dev defaults under the data directory.

Each domain ships canned cases in tests/golden/<domain>/cases.yaml: documents with declared sources, tiers, clusters and groups, plus expected verdicts. The runner executes the offline claims pipeline (so CI needs no Azure) and scores expectation pass-rate and citation precision against per-domain targets.

Recipes

Onboard a new domain (the M6 recipe)

1 · Tag existing registry sources (or add new ones) with the domain tag and per-tag tiers.
2 · Create domains/<name>/profile.yaml + entities/*.csv dictionaries.
3 · Optionally add a local_folder source pointing at corpora/<name>/.
4 · Write tests/golden/<name>/cases.yaml; run argus validate and argus golden.

Zero engine changes is the acceptance test — the shipped commodities domain is exactly this diff.

Add a fetcher adapter

Subclass Fetcher in src/argus/fetchers/, set a class-level name, implement fetch(source, ctx) → FetchResult (raise on transport failure — the poll loop isolates it), and register the class in ADAPTERS. Registry entries reference it by fetch.adapter; argus validate fails loudly on unknown names. Honor license and robots.txt.

Wire real gated discovery

Implement DiscoveryProvider.search(query, k) → [DiscoveryHit] (e.g., an Azure AI Search wrapper) and pass it as ResearchDeps.discovery. Hits become candidate proposals only — the graph never retrieves, snapshots, or cites them. Note: the CLI wiring currently hardcodes NullDiscovery (§08), so today this requires constructing deps yourself.

Custom output templates

Point output.brief_template / report_template at Jinja files (paths resolve relative to the domain directory). Variables available: domain · run_id · generated_at · window_hours (brief) or question (report) · body_md · evidence · registry_commit · synthesizer · embedder. Rendering uses StrictUndefined — typos fail fast. Keep the provenance footer.

09Web console

The same engine, without the terminal

The web console is a client of the engine, not a fourth plane: it calls the exact entry points the CLI calls — poll_domain · run_ingest · run_daily_brief · run_research · replay_run · health_report · run_golden — and adds nothing to the trust path. It exists so that operating Argus — reading briefs, running research, governing sources, watching health — needs no shell.

$ pip install -e ".[webapp]"            # fastapi · uvicorn · apscheduler · markdown
$ argus-web                              # http://127.0.0.1:8765 — run from the repo root
# offline trial: export ARGUS_LLM=stub ARGUS_EMBEDDER=hashing first (§04), or flip both in Settings
/ · Research desk

The latest brief per domain, rendered with [S#] citation chips that jump to their snapshot rows; a Run brief now button with a window override; an Ask the archive box that runs deep research and opens the report in place; recent runs and a health summary.

/admin · Governance

Four tabs: Sources (the registry — add, edit, pause / activate, per-tag tiers), Domains (profile forms in plain language: scope, trust policy, schedule, plus the entity vocabulary editor), Candidates (the review queue), Settings (§ below).

/maintenance · Operations

Six tabs: per-source Health, Schedules (per-domain on/off with next-run times), Runs (manifest viewer + one-click Replay), the searchable Snapshots archive with text previews, manual Data actions (poll · ingest · validate · golden), and the Jobs log.

09·A — How a UI save travels (AD-2 survives the mouse)

Registry and profile edits are the trust policy, so the console never writes them casually. Every save walks the same guarded path; the resulting commit hash appears in the save toast and in the provenance strip.

09·B — Jobs: one worker, on purpose

Every long action — brief, research, poll, ingest, replay, golden — runs as a background job in a single-worker queue, watched from the tray at bottom-right or the Jobs tab. One worker is a design decision, not a limitation: the dev deployment shares one SQLite file and one file-locked embedded Qdrant, so serializing all writers makes concurrent-writer bugs structurally impossible — the same reasoning as the architecture's single-process sizing. A duplicate brief for a domain already in the queue is refused (409), and a schedule tick that fires while the previous brief is still running is skipped and recorded, never stacked.

09·C — Scheduling, off by default

APScheduler runs inside the server over each profile's schedule.brief_cron — the field §07 called “documentation of cadence” is now executable. Each domain has an on/off switch in Maintenance → Schedules, off on a fresh install, persisted to data/webapp/scheduler.json; cron edits live in the domain profile (and are validated — a bad expression is a 422, not a silent dead schedule). External cron pointed at the CLI remains fully supported; pick one per domain so runs aren't doubled.

09·D — Settings without config files

The Settings tab edits every ARGUS_* knob from §07 in grouped, explained forms. Saves persist to data/webapp/overrides.env (mode 0600) and apply to the running server immediately — values saved there override the shell environment — and, because every entry point loads the same argus/envfile.py chain (defaults < .env < shell < overrides), a value saved in the console reaches argus brief under cron too. Secrets are write-only: the API reports set / not set and never echoes a value. Most settings take effect on the next run; the three bound at startup — data dir, DB URL, Qdrant URL — are flagged restart required in the UI and in the provenance strip until the server restarts.

09·E — Candidates: the loop, closed

§05·E ends with a human pasting a YAML stanza. The console keeps the same constitutional shape and removes the paste: Approve in Admin → Candidates can insert the entry directly — but it is always born paused, with license: UNREVIEWED and a placeholder endpoint, so it is structurally unfetchable until a human fills in the real feed URL and license terms on the Sources tab and activates it. Reject requires a reason and the domain is never re-proposed. Discovery still defaults to NullDiscovery in both CLI and web wiring (§08), so the queue fills only once a real provider is wired.

09·F — The provenance strip

The mono line under the masthead is the console's certification stamp, on every page: the registry commit in force (with a ·dirty flag for uncommitted edits), synthesizer and embedder mode — stub and hashing render amber so a demo configuration can never pass for production — snapshot count, queue state, missing Azure settings, and any pending restart. It is the run manifest's first line, ambient.

09·H — Access

Binds to 127.0.0.1:8765 by default (ARGUS_WEB_HOST / ARGUS_WEB_PORT). Setting a password — in Settings, applied immediately, or via ARGUS_WEB_PASSWORD — gates every page and endpoint behind a cookie session (invalidated on restart) or an Authorization: Bearer header for scripts; failed logins are delayed 0.4 s. It is a single shared secret over plain HTTP: right for a trusted machine or LAN, and anything exposed further belongs behind a reverse proxy with TLS.
